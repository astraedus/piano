### 1. What State Represents a User's Skill Level and Progression

All progression state lives in `AppState` (`src/lib/types.ts:327`), persisted under the key `practice.state` in localStorage via `src/lib/storage.ts`.

**High-level position signals:**
- `phase: Phase` (1‚Äì5) ‚Äî the coarse curriculum phase ("Learn the Layout" ‚Üí "Play Freely"). Set at onboarding, can be overridden in Settings.
- `grade: Grade` ("initial" | "g1"‚Äì"g8") ‚Äî the grade-band within a phase. Set at onboarding and in Settings. Used to tune drill selection in `computeTodayPlan`. **Never rendered to the user anywhere in the UI beyond the Settings picker itself.**
- `earLevel: 1 | 2 | 3 | 4 | 5 | 6 | 7` ‚Äî ear training difficulty ladder. Set at onboarding. Used silently in `generateEarRound` (`todayPlan.ts:120`). **Never incremented by session logic (no code in `sessions.ts` advances it), and never shown to the user anywhere in the UI.** It is a permanently static onboarding remnant unless manually edited in Settings.

**Gamification layer (V2, on top of the above):**
- `xp: number` ‚Äî lifetime XP total. Awarded by `xpForSession` in `src/lib/progression.ts`. Award amounts: session base (15‚Äì70 XP by mode), per slot engaged, ear-correct √ó 5, depth-up √ó 40, node learned √ó 100, drill quality-weighted (up to 30 + metronome/BPM bonuses).
- `level: number` ‚Äî derived from `xp` via `levelForXp`. Thresholds at 0, 100, 250, 450, 700, 1000... up to level 15+. Titles: "First Note" ‚Üí "Tinkerer" ‚Üí ... ‚Üí "Virtuoso".
- `streak: StreakState` ‚Äî `{ current, longest, lastPracticeDate }`. Forgiving: one missed day is graced (not reset). Reset only on 2+ missed days.

**Skill tree (the DAG):**
- `skillProgress: Record<string, SkillProgress>` ‚Äî per-node mastery, keyed by `SkillNode.id`. Each entry: `{ status: "locked" | "available" | "in-progress" | "learned", reps, maxBpm, firstReachedAt, learnedAt, attempts, successes, bpmReached, fluent, fluentAt }`.
- `skillReview: Record<string, SkillReviewEntry>` ‚Äî spaced-retrieval queue (R7), keyed by node ID. Each entry: `{ dueAt, intervalIndex }` on the 1‚Üí3‚Üí7‚Üí14 day ladder.
- `skillReps: Record<string, { count, maxBpm, lastAt }>` ‚Äî raw rep counters keyed by semantic slugs like `scale:C:hs`, `drill:am-pentatonic-chain`.

**Territory mastery (piano-specific):**
- `keyDepths: Partial<Record<KeyId, KeyDepth>>` ‚Äî per-key depth (0‚Äì5): Untouched / Heard / Walked / Played / Lived / Home. 24 total keys.

**Content progression:**
- `unlocks: UnlockCard[]` ‚Äî earned capability cards (with `addedAt` timestamps).
- `pendingUnlocks: UnlockCard[]` ‚Äî queued to show post-session.
- `pieces: Piece[]` ‚Äî song shelf entries, each with `status: "learning" | "shelved" | "yours" | "known"` and total `minutes`.
- `arc: ArcEvent[]` ‚Äî append-only timeline of significant moments: `instrument-begins`, `phase-begins`, `unlock`, `piece-yours`, `first-improv`, `piece-started`, `level-up`.

**Learning path:**
- `learningPath?: PathTag` ("just-play" | "play-with-soul" | "go-deep") ‚Äî chosen intent, set at onboarding. `undefined` = show everything (legacy compat).
- `theoryEnabled?: boolean` ‚Äî whether theory nodes render in the graph.

### 2. What Is Persisted to Neon Per-Account vs. Local Only

**Architecture:** `localStorage` (`practice.state`) is the live working copy. Neon (`practice_state` table keyed by Clerk `userId`) is the durable account save. They store **the same blob** ‚Äî the full `AppState` JSON.

**Neon sync model (`src/lib/cloudSync.ts`, `src/components/CloudSyncManager.tsx`):**
- Pull once per browser session on sign-in: if cloud differs from local ‚Üí adopt cloud + reload; if cloud empty ‚Üí seed with local.
- Auto-push: debounced at 1500ms on any state change. Push = full `AppState` blob (`JSON.stringify(state)`).
- Manual escape hatches in Settings page via `CloudSync.tsx`: "Sync now" (force push) and "Restore from cloud" (force pull + reload).

**What is in Neon:** Everything in `AppState` ‚Äî every field listed above. There is no selective sync; it is one JSONB column.

**What is local-only (never in Neon):**
- `sessionStorage.getItem("pianoCloudPulledSession")` ‚Äî the per-tab pull flag (not in AppState, lives in sessionStorage, ephemeral).
- Per-drill rep session snapshots (`localStorage.getItem(repSessionKey(...))`) ‚Äî the in-flight rep engine state (`src/components/slots/repResume.ts`). These are written to separate localStorage keys (not part of `practice.state`), so they are never pushed to Neon.

**Auth signal:** `ProfileChip` (`src/components/ProfileChip.tsx`) shows signed-in state (green dot + Clerk `UserButton`) vs anonymous (grey "Anonymous" pill that triggers sign-in modal). It appears in the top-right of the persistent header, always visible.

# Piano App ‚Äî Learning Loop & Content Audit

## 1. Walk of a Single Practice Session

**Entry point:** `PracticeStand` renders after `computeTodayPlan` (`lib/todayPlan.ts:computeTodayPlan`) resolves the session config from persisted `AppState`.

### What the plan computes before the user touches anything

| Decision | Where | Result |
|---|---|---|
| Ghost key (key of the week) | `lib/ghostKey.ts:ghostKeyFor` | One key per week, deterministic from state |
| Gap check | `todayPlan.ts:gapDaysSince` | Sets `mode = "first-back"` if ‚â•3 days absent |
| Warmup | `todayPlan.ts:warmupForWeek` ‚Üí `piano/warmups.ts` or `guitar/warmups.ts` | Rotates 5‚Äì7 warmup types by week/phase |
| Chain drill | `chainDrillPicker.ts:pickChainDrill` | Phase-filtered, ghost-key-preferred, recent-5 avoided, day-of-year seeded |
| Interleave | `chainDrillPicker.ts:buildInterleavePlan` | Weaves 2‚Äì3 established drills if eligible |
| Ear rounds | `earRounds.ts:generateEarRound` √ó 3 | Level-gated: L1=maj/min, L2=scale degree, L3=quality, L4=cadence, L5=progression |
| Review skills | `skillReview.ts:dueReviews` | Spaced-repetition queue of already-learned nodes |

### The five slots, in order

**Slot 1 ‚Äî Warmup** (`WarmupSlot.tsx`, pillar: `technique`)

Duration: ~90 seconds. Rotating exercise keyed to this week's ghost key ‚Äî e.g. "ghost-scale" (hands-separate scale + keyboard visual + "Hear the Scale" audio button), "weight-transfer" (5-finger pattern), "triad-tour" (I‚ÄìIV‚ÄìV‚Äìvi walk), "mirror" (hands-contrary), "free", "parallel-sets", "tone-drill". The user taps "I Played It" which calls `bumpRep(scaleRepId(ghostKey))` ‚Äî that rep counter is the slot's done-gate. A metronome is available. Progress persists: "X reps of this scale ¬∑ best Y BPM."

**Slot 2 ‚Äî The Piece** (`PieceSlot.tsx`, pillar: `repertoire`)

No structured content: user-entered piece (title, composer, key, sheet URL, reference recording). If a YouTube link is stored, an iframe embeds inline. The only "learning" here is what the user brings. No instructions, no practice method is suggested. The slot is a tracking surface for a self-chosen piece.

**Slot 3 ‚Äî Chain Drill** (`ChainDrillSlot.tsx`, pillar: `improv` display-wise, but `drill.pillar` varies)

This is the app's richest slot. Structure:
1. "Tonight" lesson block ‚Äî `getLesson(instrument, drillNode.id)` (`lessons.ts`) shows `what` / `why` / first `step`. Links theory terms via `linkTerms()` ‚Üí tappable `TermChip` glossary cards.
2. **RepEngine** (`RepEngine.tsx`) ‚Äî interactive rep loop:
   - Counts "Rep N of M"
   - "Clean rep" / "Missed it" buttons per rep; `lastWasSuccess` flash ("Locked in.")
   - If `repBlocks` defined: enforced micro-rest countdown between blocks ("Rest. This is when it sticks.")
   - If `bpmLadder` defined: BPM display + ladder progress bar; after 3 consecutive clean reps prompts "Bump to N BPM?"
   - If `interleave` defined: skill-switch cue animates, "Now: [drill name]"
   - Session persists to `localStorage` (day-scoped), resumes on reload
3. "Show steps" disclosure ‚Äî numbered step list with durations
4. "Hear it ¬∑ see the shape" disclosure ‚Äî progression on keyboard/fretboard visual + "Hear the Loop" + "Hear Pentatonic" audio buttons

The drill content itself is a 4‚Äì6 step `ChainStep[]` sequence baked into the data files:
- Piano Phase 1: scale ‚Üí triad ‚Üí I‚ÄìIV‚ÄìV‚ÄìI progression ‚Üí pentatonic improv ‚Üí song reference
- Guitar Phase 1: technique reps (tone/scale) ‚Üí chord shapes ‚Üí progression changes ‚Üí riff/song
- Every drill ends with a real song reference ("You just played blues.")

**Slot 4 ‚Äî Ear Moment** (`EarMomentSlot.tsx`, pillar: `ear`)

3 rounds √ó ~20 seconds each, with 8-second rest between. The audio plays, user picks from 2‚Äì4 choices. Correct/wrong tracked and folded into `earResults` on session end (feeds XP and ear level progression). Levels gate the round type: level 1 = major-vs-minor, level 2 = scale degree identification, level 3 = chord quality (maj/min/dim/aug), level 4 = cadence type (V-I, IV-I, ii-V-I), level 5+ = progression identification (4 pop/jazz patterns). Tappable `TermChip` explainers appear beneath the answer choices.

**Slot 5 ‚Äî Free Play** (`FreeSlot.tsx`, pillar: `expression`)

Minimal structure: three bullet prompts ("Play anything. Jump back into your piece. Pick up something you heard today."), a saved URL field (MuseScore/YouTube link), and a one-line journal. If `reviewSkills` contains due nodes, a "Bring these back" card surfaces each learned node's `masteryDrill` with a "Did It" button that advances the spaced-review interval.

**Session end (`handleDone`)**

Calls `endSession(state, logBase, date)` in `lib/sessions.ts`. That function:
- Bumps key depth for the ghost key (and piece key if set)
- Marks skill nodes `learned` when their linked `chainDrillId` was played AND all prereqs are `learned` AND success rate ‚â• 70% (`meetsLearnSuccessRate`)
- Fires `UnlockCard` modals for newly learned nodes
- Computes XP (base + per-slot + per-ear-correct + per-node-learned + quality-weighted drill XP)
- Updates streak and level; fires `LevelUpModal` if level crossed

**Why it's the weakest:**

**Depth:** The ear system only runs three rounds per session, all identical in structure (play audio ‚Üí pick label). There is no progressive difficulty within a session ‚Äî L1 major/minor is trivially easy, and the ladder requires playing many sessions to climb. Levels 6‚Äì7 have no distinct round type yet (`earRounds.ts:generateEarRound` fallback `progressionRound(ghostKey)` on both). The ear is never asked to *sing*, *reproduce*, or *anticipate* ‚Äî only to *label after hearing*.

**The theory-to-practice gap is fully passive.** `TermChip` is superb as an encyclopaedia, but theory is only accessed when the user taps a blue chip. Nothing in the session flow actively tests "can you recognize a ii‚ÄìV‚ÄìI when you hear it" or "which scale degree is this riff starting on?" The `EarMomentSlot` and the `TermChip` live in separate streams that never intersect.

**Shape-spotting for guitar (fretboard patterns) is absent.** The `Fretboard` component exists as a display (`lib/guitar/components/Fretboard.tsx`), and `GuitarMap.tsx` exists as a component, but there is no exercise that asks "where does this pentatonic box live on the neck?" or "find the root of this chord shape." Fretboard pattern recognition is a distinct skill for guitar ‚Äî the CAGED system is partially in the skill node data (`cagedShape?: "E"|"A"|"G"|"C"|"D"` on nodes) but has no training loop attached.

**Transcription is buried in Phase 2+.** The two transcription drills (`p1-echo-ear` and `p2-song-transcribe`) require the skill node to already be reached in the DAG. A Phase 1 player gets the Ear Moment slot but no "play what you hear" prompt ‚Äî the echo exercise is locked behind `p-t1-echo-ear` reaching `available` status.

**No contour/interval training.** There are no interval-identification rounds (major 3rd, perfect 5th, etc.) despite `EarRoundType` having `"interval"` defined in `lib/types.ts:EarRoundType`. That round type exists in the type union but `generateEarRound` never generates one.

## 4. Where New Mechanics Plug In

The architecture is cleanly extensible. Every insertion point below is additive ‚Äî no core rewrites needed.

### A. Interactive Ear Slot (richer Pattern Recognition within existing `EarMomentSlot`)

**Plug point:** `lib/earRounds.ts:generateEarRound` + `EarRound` type in `lib/types.ts`.

Add new `EarRoundType` variants and new level-gated generators:
- `"interval"` ‚Äî play two notes, name the interval (the type already exists in the union at `types.ts:EarRoundType`)
- `"sing-back"` ‚Äî "Hear the tonic, then a note. Sing (or find) it on the instrument." Could use the existing `EarMomentSlot` flow with a "Did you get it?" honour button instead of multiple choice
- `"fretboard-spot"` (guitar) ‚Äî given a played note/chord, which box/position is it in? `EarChoice` labels could be fret-range labels; `seeChordShape` on the glossary entry could cross-link

No new slot needed; all variants consume the same `EarRound` interface and render through `EarMomentSlot`.

### B. Guitar Shape-Spotting Round (Pattern Recognition for guitar)

**Plug point:** `lib/guitar/earRounds.ts` (already exists as a separate guitar ear module, already wired via `todayPlan.ts:earRoundFromModule`).

`guitar/earRounds.ts:generateGuitarEarRound` can return rounds whose `audio.kind` triggers the fretboard visual (`seeKind: "fretboard"` in glossary terms). The slot already handles `EarRound` generically ‚Äî a guitar-specific round with "which box is this scale position?" just needs a new generator function and a `level` gate. The `GuitarMap` component (`components/GuitarMap.tsx`) is already built and could be surfaced in the `EarMomentSlot`'s "See" context.

### C. Structured Improv Prompts (Generation within the Chain Drill)

**Plug point:** `ChainStep.type` in `lib/types.ts`. The union already has `"improv"` but drill steps only carry plain text instructions.

New step types or enriched step data could add:
- `"improv-prompt"` with a `{ startNote: string; endNote: string; bars: number; constraint: string }` payload ‚Äî the slot renders an interactive prompt card instead of just text
- A "phrase bank" button in `ChainDrillSlot` for the improv step (random rhythmic or melodic constraint: "use only 3 notes", "start on the 5th", "end on rest") ‚Äî zero backend needed, pure client-side RNG

The `ChainDrillSlot` already has a progressive-disclosure pattern (`<Disclosure>`) that could host an improv scaffold without crowding the main rep flow.

### D. Melodic Dictation / Transcription Loop (Generation + Pattern Recognition together)

**Plug point:** New drill type or new slot. The cleanest fit is a new `ChainStepType` variant ‚Äî `"dictate"` ‚Äî where the step plays a melodic fragment via `playSequence` (already exists in `lib/audio.ts`) and the user finds it on the instrument, then taps "Got it / Not quite" as an honour system.

The `audio.ts:playEarRound` function already handles `scale-degree` playback, so the audio infrastructure exists. The `EarMomentSlot` could grow a "find it" mode triggered when an `EarRound` has `type: "scale-degree"` and `level >= 3`.

### E. Theory-Active Prompts (connecting TermChip knowledge to the Ear slot)

**Plug point:** `EarChoice.termId` in `lib/types.ts` ‚Äî already used to link ear choices to glossary entries (`earRounds.ts` uses this on the `major-vs-minor` and `pop-formula` entries).

Extend this: after a cadence round answer, instead of just "correct/wrong" text, the feedback could auto-trigger the glossary entry for the correct choice ("V‚ÄìI is a perfect cadence. Hear it again."). The `TermChip` component + `useExplain` context (`components/explain/useExplain.tsx`) already manage the open/close state. An `onAnswered` callback in `EarMomentSlot` could programmatically open the correct `TermChip` glossary card ‚Äî closing the theory-to-practice loop without any new data infrastructure.

### F. Progress Visibility ‚Äî Progression Dashboard (the "how far you've come" ask)

**Plug point:** `/tree` page (`app/tree/page.tsx` ‚Üí `SkillGraphView` / `SkillGraph`), XP + streak in `PracticeStand.tsx:StatsStrip` and `XPBar`.

Progression is tracked in full detail but surfaced sparsely:
- `XPBar` and `StreakFlame` appear in the desktop info rail only (hidden `<aside className="hidden lg:block">` in `PracticeStand.tsx:259‚Äì260` ‚Äî wait, actually it is not `hidden lg:block` on the aside itself ‚Äî the aside is part of the grid on `lg+`, but `StatsStrip` inside it only has `mt-4 flex` with no explicit hide)
- The skill graph at `/tree` is the real progress view, but it is not linked from the session end screen
- `session line` at Done (`doneLineFor`) celebrates the specific session but doesn't connect to overall arc

New mechanics plug in cleanly: after `handleDone` in `PracticeStand`, the existing `sessionLine` text could be augmented with a link to the skill tree filtered to "newly unlocked" nodes. `newUnlocks` is already returned by `endSession` and queued in `unlocksQueue` ‚Äî these could also surface a "see your path" CTA that routes to `/tree?highlight=<nodeId>`.
